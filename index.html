A plain-language guide to the product

Governance an AI agent
can't switch off.

AEG — Auto Engine Governance — is a fail-closed governance envelope for AI agents: a supervisor that sits between an AI agent and the real world and lets it do only what it has been explicitly, verifiably authorized to do — recording every action so it can't be quietly changed later.

Explore the pipeline → How it works
The one principle
Fail-closed: if anything is missing, unverified, tampered with, or even just uncertain, the answer is "no".
A bank vault door that loses power stays locked, not open. Every part of AEG works the same way — permission is never assumed; it must be proven, every time.
The core idea
Picture the AI agent as a skilled worker in a locked room — no keys, no tools, no outside phone line — with one service window.
The agent owns nothing: not the codebase, not the passwords, not the internet connection. AEG holds all of it. To do anything at all, the agent slides a request through the window, and the gateway performs the action only if the signed rulebook allows it. Because the agent has no other way to act, governance can't be bypassed or switched off. In the product this is the V2.3 gateway — a possession-based execution broker.
How each part works

The technical names are kept so they line up with the product's own documents; each is explained in everyday terms, with a familiar comparison.

Signed boundaries the rulebook

Before the agent may do anything, someone with authority issues a boundary — a precise, signed instruction spelling out which commands may run, which files may be touched, and who may approve risky steps. The agent cannot widen it, edit it, or write its own.

Like a notarized permission slip — a signed document with an unforgeable seal, not a friendly suggestion.

The task itself where the job comes from

The instruction the agent must carry out doesn't arrive on a separate, ungoverned channel. It travels inside the same signed boundary — a task field with a description (what to do) and a rationale (why) — right next to the scope that fences it and the criteria that define "done".

Like a sealed work order that states the job, the rules, and what "finished" looks like — all on one notarized page.

Checking the seal fail-closed verification

Every time a boundary is used, AEG verifies its cryptographic signature. If the seal is missing, broken, or from an issuer it doesn't trust, the boundary is rejected and nothing happens.

Like a border officer checking a passport's security features, not just glancing at the photo. No valid seal, no action.

Two-key approvals quorum

Actions too sensitive for one person stay blocked until the required number of distinct, verified approvers sign off. Proven live: with one approval the action was denied; with two real, independent identities it went through.

The two-key nuclear launch — or a large wire transfer that needs two officers to both sign.

No self-approval separation of duties

Whoever requests an action cannot also count as one of its approvers, so the two-key rule can't be satisfied by one person wearing two hats.

The employee who submits an expense report isn't allowed to approve it.

The supervised hands governed execution

Allowed commands and file writes go through AEG's governed path; the agent never touches the system directly. Anything outside the boundary is denied, and the record shows it never ran — including the common trick of hiding a forbidden action inside an innocent one ("just run this little script for me").

A worker who must hand every request to a supervisor, who won't pass along a smuggled instruction.

The tamper-evident logbook audit chain

Every governed action is written to a log where each new entry carries a cryptographic fingerprint of everything before it. Alter any past page and every seal after it breaks — the tampering is obvious at once.

The same chaining a blockchain uses, applied to an audit trail: a numbered logbook you can't quietly rewrite.

The notary's timestamp RFC 3161 anchoring

The sealed logbook is anchored to a trusted external timestamp authority, so no one can secretly rebuild a fake history after the fact. Proven against a real, public authority.

An independent notary stamping the official date and time onto the record.

The honesty inspector non-vacuity validator

When the agent claims a job is "done," a deterministic six-check validator examines the claim and refuses a success that did nothing meaningful. An empty "everything passed" is treated as a failure, not a pass.

A building inspector who won't sign off just because every checkbox is ticked — if no real work was verified, the sign-off is refused.

The sealed evidence box offline bundles

A whole governed run can be packaged into one evidence bundle an auditor opens on a completely disconnected laptop and re-verifies themselves — no trust in our servers, the vendor, or the internet. Alter one byte and verification fails and points to exactly what changed.

A sealed evidence box anyone qualified can open and check independently.

The clean-room check attestation-gated sessions

Before a session opens, the sandbox must prove what it is running — a measured, approved environment answering a fresh, single-use challenge from the gateway. No verified attestation, no session; the door stays shut by default.

Like competitive anti-cheat that refuses to start the game unless the environment is provably clean.

Real identities trust roots

Approvers sign in with their real corporate identity, producing a signature tied to a genuine person and recorded in a public transparency log — not a test credential.

Like "Sign in with Microsoft or Google" — the same family of technology (Certificate Transparency) that keeps website certificates honest.
