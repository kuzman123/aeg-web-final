Workflow loop

How a task flows from human approval to evidence and back to a human decision.

• Human Principal

→

←

Human approval defines scope. Local evidence supports review.
The tool records what passed through its governed path; the human at both ends owns approval and acceptance.
Accuracy note. The tool records and checks; it does not approve. The Human Principal at the end of the loop decides whether the evidence supports a done claim — a positive Evidence Report is not a green light by itself.

    Visual 04   Unavoidable by design

    Unavoidable by design

    Governance the agent can't opt out of — every path to action runs through the envelope.

    Inside the envelope enforced, fail-closed
    Every governed command & write
    Runs only if a signed boundary allows it; denials are recorded.
    Two-key approvals (quorum)
    Sensitive actions need distinct verified approvers; you can't approve your own.
    Tamper-evident audit + anchor
    Merkle-chained log, RFC 3161 time-anchored, offline-verifiable bundle.
    How possession makes it unavoidable the architecture
    The gateway owns everything
    Repo, credentials, and egress live with AEG; the agent holds none, so it must request every action.
    One governed path
    Commands, writes, and reads all flow through the same boundary → quorum → execute → audit path.
    Attested & physically separated
    Sessions open only against a verified attestation of a measured sandbox — proven on two physical machines: the agent in a Firecracker microVM booted from a measured image, the gateway holding repo, credentials, and egress on its own hardware.
    Governance by possession. Because the gateway holds the repo, credentials, and egress, the agent cannot act except through the envelope — supervision is a property of the architecture, not a policy the agent can ignore.
    Visual 05   Boundary record & the task

    The boundary record & the task

    One signed record carries the task the agent must do, the scope that fences it, and what counts as done. There is no separate, ungoverned task channel.

    boundary.signed.intoto.jsonlcosign / Sigstore signed
    {
      "boundary_id": "b-2026-07-06-gateway",
      "task": {
        "description": "Add a CSV export endpoint with tests",
        "rationale":   "customer reporting requirement"
      },
      "scope": {
        "files":    { "allowed_paths": ["src/**","tests/**"], "forbidden_paths": [".env",".git/**"] },
        "commands": { "allowed": ["pytest","git"], "allow_interpreters": false }
      },
      "approval_quorum": { "min_required": 2, "approvers": ["oidc:…","oidc:…"] },
      "acceptance_criteria": [
        { "criterion_id": "ac-1", "text": "endpoint returns 200 with header row", "evidence_type": "test_pass" },
        { "criterion_id": "ac-2", "text": "docs/reports.md updated", "evidence_type": "file_exists" }
      ],
      "done_claim_policy": { "require_anchor": true, "co_signer": true }
    }
    How the task reaches the agent
    The task lives in the boundary
    The Human Principal writes task.description (what to do) and task.rationale (why) into the same predicate as the scope, then signs it.
    Scope fences the work
    Allowed / forbidden paths and commands bound what the agent may touch while carrying out the task.
    Criteria define "done"
    Each acceptance_criteria entry ties a requirement to concrete evidence — a test pass, a file, a command output, or a manual attestation.
    No side channel
    A session opens only against a verified boundary; the agent then acts through the tools, and its done-claim is validated against these criteria. The instruction and its guardrails are one signed unit.
    Illustrative values; real fields. The boundary-v1 predicate's required fields include task, scope, acceptance_criteria, approval_quorum, and done_claim_policy — so the task and its guardrails are one signed, tamper-evident record. Signatures resolve to real corporate identities via cosign / Sigstore OIDC.

    Visual 07   Audit chain & log validation

    Audit chain & log validation

    Tamper-evident logging over the action log — hash chain, external anchoring, and offline verification.

    Hash chain
    SHA-256 continuity over log records. The default; detects reorder, insertion, and deletion.
    RFC 3161 anchor
    An external timestamp authority signs the chain root — a third-party time proof the local host cannot forge.
    Offline bundle
    aeg-verify re-checks signatures, chain, and anchor on a disconnected machine.
    Record types · log-validate
    v1_chain_only
    Chain continuity
    v2_dsse
    DSSE · local-verified
    v1_hmac
    HMAC · external key required
    migration_v1_to_v2
    Migration · structural
    status: valid — hash-chain continuity checked. SHA-256 chain over records; the default, no external key material needed.
    .auto_engine/log.jsonl · log-validate
    #ACTIONPREV_HASHHASH
    1run-command pytest…0000…7a1f…b2ok
    2write-file src/util.py7a1f…b2c20e…91ok
    3run-command "rm -rf /" (rej)c20e…91e84b…1fok
    4claim-donee84b…1f13af…7cok
    ✓ chain intact (4 records, SHA-256)
    record reordersilent insertionsilent deletion
    External RFC 3161 anchoring seals the chain root — a third-party time proof that even an attacker with log write-access cannot forge.
    Tamper-evidence you can check independently. A SHA-256 hash chain over every record, an RFC 3161-anchored root, and offline bundle verification with aeg-verify — an auditor confirms the log was not altered without trusting the host that produced it.
